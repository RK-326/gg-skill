# Дизайн-спека (эталон для всех документов)

ВСЕ документы портала выглядят ИДЕНТИЧНО эталону. Здесь зафиксированы точные
значения, чтобы дизайн получался с первого раза.

Эталон оформления (живой файл-образец): `template/example.docx`
(копия «Бакалавриат Инструкция - обновлено.docx»). Пустая болванка с теми же
стилями/колонтитулами/водяным знаком — `template/blank.docx`.

Технические единицы: размеры шрифта в OOXML в half-points (sz=22 → 11pt).
EMU: 914400 EMU = 1 дюйм.

## Страница и поля
- Размер: Letter, `<w:pgSz w:w="12240" w:h="15840"/>` (8.5×11").
- Поля: `<w:pgMar w:top="576" w:right="432" w:bottom="0" w:left="432" w:header="720" w:footer="720" w:gutter="0"/>` (узкие: top 0.4", left/right 0.3").
- Полезная ширина текста ≈ 7223760 EMU — это ШИРИНА всех плашек-shape (`cx="7223760"`).

## Шрифты (все Montserrat, шрифты встроены в файл)
- Боди/по умолчанию: **Montserrat Medium**, sz 22 (11pt), межстрочный 276, spacing after 200.
- Заголовок документа (Title): **Montserrat Black**, sz 60 (30pt), letter-spacing `<w:spacing w:val="5"/>`, spacing after 300.
- Раздел (Heading1): **Montserrat** + `<w:b/>`, sz 36 (18pt), spacing before 480, `<w:pageBreakBefore/>` (каждый раздел с новой страницы), `<w:outlineLvl w:val="0"/>`.
- Подзаголовок (Heading2): тема majorHAnsi + `<w:b/>`, sz 26 (13pt), цвет `4F81BD`.

## Цветовая палитра (HEX, всегда srgbClr / w:color)
- Боди-текст и заголовки 1 уровня: **13445D** (фирменный тёмно-синий, цвет по умолчанию в docDefaults). ВЕСЬ текст в таблицах тоже 13445D (не 202A35, не чёрный).
- Акцент / маркеры списков / номера / подчёркивания в оглавлении: **1A5CA8**.
- Заголовок 2 уровня: **4F81BD**.
- Шапка таблиц: заливка **2C3E50**, текст **FFFFFF** (детали — `../tables/tables-standard.md`).
- Рамки (таблицы, картинки, светлые карточки): **D5DBDB**, толщина `w="9525"` (≈0.75pt), `single`.
- Плашка ВАЖНО: заливка **13445D**, заголовок «ВАЖНО» белый **FFFFFF**, текст **DCE6EC**.
- Плашка «Коротко по разделу»: заливка светло-зелёная **EAF6EF**, заголовок **2C3E50**, текст 13445D.
- Светлая инфо-карточка («Как устроен этот гайд», врезки): заливка **F4F6F7**, рамка D5DBDB, заголовок 13445D.
- Колонтитул: «CONFIDENTIAL» серый **D0D0D0**; бейдж номера страницы синий (1C6083/0B2E40), цифра белая.

## Тень (ОДНА на весь документ, у всех картинок и плашек)
```xml
<a:effectLst><a:outerShdw blurRad="50800" dist="25400" dir="5400000" rotWithShape="0">
  <a:srgbClr val="202A35"><a:alpha val="30000"/></a:srgbClr></a:outerShdw></a:effectLst>
```
Для картинок тень мягче: dist="19050", alpha="24000". 202A35 здесь — цвет ТЕНИ, не текста.

## Плашки = shape `wps:wsp` (НЕ таблицы). Общий рецепт
Inline-drawing, `<a:prstGeom prst="roundRect"><a:avLst><a:gd name="adj" fmla="val 7000"/></a:avLst></a:prstGeom>`, ширина `cx="7223760"`, `<a:ln><a:noFill/></a:ln>` (у светлой карточки вместо noFill — рамка D5DBDB w=9525), тень (см. выше), `wps:bodyPr` с отступами `lIns="200000" tIns="150000" rIns="200000" bIns="150000"` и `<a:spAutoFit/>`. Внутри `wps:txbx/w:txbxContent` — обычные `w:p`.
- **ВАЖНО**: `solidFill 13445D`; п.1 = `<w:b/><w:color w:val="FFFFFF"/>` sz21 «ВАЖНО»; п.2 = `<w:color w:val="DCE6EC"/>` sz21 текст.
- **Коротко по разделу**: `solidFill EAF6EF`; заголовок `<w:b/><w:color w:val="2C3E50"/>` sz21; пункты с маркером «•».
- **Светлая карточка**: `solidFill F4F6F7` + `<a:ln w="9525"><a:solidFill><a:srgbClr val="D5DBDB"/></a:solidFill></a:ln>`, adj можно 5000.

## Картинки и скриншоты (ВСЕ одинаково)
В каждом `pic:spPr`: геометрия `roundRect` adj 4200 + рамка `<a:ln w="9525">` D5DBDB + тень (dist 19050, alpha 24000). Порядок детей spPr: xfrm, prstGeom, (fill), ln, effectLst. Скриншоты НЕ оставлять прямоугольными без рамки. Текстовый контент НЕ держать картинкой — переносить в обычный текст/маркеры. Стиль генерируемых картинок — `../images/README.md`.
- ЕДИНАЯ ШИРИНА всех full-width элементов — ЖЁСТКОЕ ПРАВИЛО (брак, если нарушено): hero-картинка,
  карточка-таймлайн (roadmap «Дорога к поступлению»), карточка «Как устроен» (TOCBox), плашки
  (ВАЖНО / Коротко по разделу / светлые карточки), ВСЕ инфографики-баннеры (схема GPA, бюджет-чарт,
  анатомия эссе и любые сгенерированные иллюстрации), а также ТАБЛИЦЫ — ВСЕ ровно `cx="7223760"` EMU
  (таблицы — `tblW pct 5000` = 100% ширины текста, сумма gridCol ≈ 11376 twips). Левые и правые края
  ВСЕХ этих элементов обязаны совпадать в одну вертикаль по всему документу. Картинка/инфографика
  НЕ должна быть у́же остальных: любую full-width картинку приводить к 7223760, высоту пересчитывать
  пропорционально пикселям (`cy = round(7223760 * h / w)`), БЕЗ горизонтального растяжения.
  ИСКЛЮЧЕНИЕ (намеренно НЕ full-width): фото автора (узкое, по центру) и парные/тройные UI-скриншоты
  навигации — они стоят рядом в одном абзаце и в сумме дают ~полную ширину, поэтому каждый по
  отдельности у́же. Всё ОСТАЛЬНОЕ — одинаковой ширины 7223760.
  Частые баги: таймлайн бывает у́же (6858000 / 5918299); инфографики (GPA 3938625, бюджет 5120640)
  собираются у́же hero — ОБЯЗАТЕЛЬНО выровнять все до 7223760, иначе края «гуляют» (см. фидбэк Ренаты).

## Раскладка картинок и пустоты (эталон = магистратура, копировать 1-в-1)
Документы портала имеют ОДИНАКОВУЮ раскладку. Где контент совпадает с уже выверенным
документом (магистратура) — копировать его раскладку буквально, не изобретать.
- ВСЕ картинки и скриншоты — INLINE (`wp:inline`), НИКОГДА не плавающие (`wp:anchor`/`wrapSquare`).
  Плавающие заставляют текст обтекать картинку и рвут предложения («Вот, как [картинка]
  университетские клубы…»), гуляют по странице и оставляют дыры. В эталоне 0 плавающих.
  Любую `wp:anchor` конвертировать в `wp:inline` (оставить extent/effectExtent/docPr/
  cNvGraphicFramePr/graphic, выкинуть simplePos/positionH/positionV/wrap*).
- Текст и картинки НЕ в одном абзаце. Подпись-лид («…можно найти так:», «Пример вкладки …:»)
  — отдельный абзац; СКРИНШОТЫ — следующий ОТДЕЛЬНЫЙ абзац: `jc=center`, `spacing before=80 after=120`.
  Парные/тройные UI-скрины — в одном таком абзаце рядом (горизонтально). Так появляется
  воздух между текстом и скринами и они не липнут.
- НЕТ прогонов пустых абзацев. Максимум ОДИН пустой абзац подряд (как в эталоне). Прогоны
  из 5-10-28 пустых абзацев = гигантские дыры вверху страниц и переносы картинок на след.
  страницу — схлопывать до одного (не трогая абзацы с `sectPr`/`pageBreakBefore`).
- Титульный блок (стр. 1) умещает: заголовок + подзаголовок + roadmap-баннер + строку
  «Данные проверены …» + hero-картинку + карточку «Как устроен» — ВСЁ на одной странице.
  Если hero уехал на стр. 2 — обычно roadmap слишком высокий: он должен быть ПЛОСКИМ
  баннером (как у маги: 2400×540 px, при `cx=7223760` → `cy=1625346`). Чанковый roadmap
  (аспект ~3:1, cy≈2.3 млн) выталкивает hero — перегенерировать плоским тем же
  `scripts/`-стилем (заголовок DK 13445D, этапы BLUE 1A5CA8 bold, синяя линия с точками,
  подписи серым), сохранив подписи этого документа. БЕЗ лишней маленькой «лид-точки» перед
  первым узлом — линия начинается прямо с первого кружка-этапа (иначе выглядит как случайный буллет).

## Списки-маркеры (обычный текст, без боксов)
Обычный `w:p`, `<w:spacing w:after="20"/>`, текст начинается с литерала «•  » (маркер можно цветом 1A5CA8). Никаких декоративных боксов вокруг простых перечислений.
- МАРКЕР ВСЕГДА круглый «•», НИКОГДА стрелка «➜» (U+279C) и никакие другие стрелки/символы.
  Это касается и буллет-списков через `numbering.xml`, И ЛИТЕРАЛЬНЫХ стрелок прямо в тексте —
  особенно в ПЛАШКАХ «Коротко по разделу» (там пункты часто начинаются литералом «➜ » в run-тексте,
  numbering их не трогает). Заменять «➜» на «•  » в `document.xml`. (Фидбэк Ренаты: «убрать стрелочки
  ВЕЗДЕ, включая Коротко по разделу».) Перед сдачей: `grep` document.xml на ➜/→ = 0.
  ГОЧА: в `numbering.xml` мог затесаться `abstractNum`, где `<w:lvlText w:val="➜"/>` со шрифтом Arial —
  это и есть стрелка-маркер. Заменять на круглый буллет: `w:val=""` (символ ) + `rFonts ascii/hAnsi/cs="Symbol"`,
  как у остальных bullet-списков. Проверять numbering.xml на отсутствие `➜` перед сдачей.
- ЦВЕТ И ВЕС МАРКЕРА (фидбэк Ренаты): буллет тёмно-синий **13445D** (как текст) и **bold** —
  в `<w:lvl>/<w:rPr>` добавить `<w:b/>` и `<w:color w:val="13445D"/>`. Для ВСЕХ bullet-уровней.
- ГОЧА ПУСТОГО МАРКЕРА: U+F0B7 невидимый (Private Use); при правке скриптом вставленный «как есть»
  символ обнуляется → `w:val=""` пустой → маркер ПРОПАДАЕТ совсем (так сломалась маг v29).
  Задавать ТОЛЬКО явным escape ``, после сборки проверять `len(lvlText.val) > 0` у всех уровней.
- Списки на стиле: абзацы часто имеют `pStyle="ListBullet"` без своего `numPr` — маркер берётся
  из стиля → `numId` → `abstractNum`, поэтому править надо `numbering.xml` (не каждый абзац).

## Сноски
Настоящие сноски Word (`footnotes.xml`), стиль FootnoteText sz20. Если ДВЕ сноски подряд — между ними запятая отдельным надстрочным ран'ом:
`<w:r><w:rPr><w:rStyle w:val="FootnoteReference"/><w:vertAlign w:val="superscript"/></w:rPr><w:t>,</w:t></w:r>` → выглядит как «⁴,⁵».

## Водяной знак (в header1.xml, на каждой странице)
Логотип GG (VML `WordPictureWatermark1`, imagedata), washout `gain="42598f" blacklevel="13107f"` (заметный, но фоновый). ОБЯЗАТЕЛЬНО по центру страницы:
`style="...;margin-left:0;margin-top:0;width:455pt;height:374pt;mso-position-horizontal:center;mso-position-horizontal-relative:margin;mso-position-vertical:center;mso-position-vertical-relative:margin"`.

## Карточка «Как устроен этот гайд» (точечный заполнитель)
Строки оглавления-карточки используют правый таб с точечным заполнителем
(`<w:tab w:val="right" w:leader="dot" w:pos="10350"/>`) и номером страницы справа.
ПРАВИЛО (баг, который нельзя допускать): текст строки (Раздел N + тема + описание)
должен оставлять место для заполнителя. Если описание слишком длинное, текст
упирается в правый край, точек почти нет, номер прижимается к тексту и ломается
симметрия с другими строками.
- Держать видимый текст строки коротким (ориентир: не длиннее самой короткой из
  соседних строк; до правого таба оставлять заметный зазор под точки).
- Если тема не влезает - сокращать описание (а не множить точки и не упирать номер).
- Перед сдачей визуально сверять: у ВСЕХ строк карточки одинаковая дорожка из точек
  и номера выровнены в один столбец справа.

## Колонтитулы
- Footer: слева «CONFIDENTIAL» (D0D0D0), справа бейдж с номером страницы (поле PAGE, синий бейдж 1C6083, белая цифра).
- ПОЗИЦИЯ номера страницы: ОДИН правый таб-стоп по правому полю текста
  (`<w:tabs><w:tab w:val="right" w:pos="ШИРИНА_ТЕКСТА"/></w:tabs>`, где ШИРИНА_ТЕКСТА =
  pgW − left − right, для Letter с полями 432 = 11376). Между «CONFIDENTIAL» и бейджем
  ровно ОДИН `<w:tab/>`. НЕЛЬЗЯ толкать номер вправо пачкой табуляций «на глаз» —
  тогда он съезжает и несимметричен. Бейдж должен стоять в нижнем правом углу,
  зеркально «CONFIDENTIAL» слева.
- ВНУТРЕННИЕ ОТСТУПЫ бейджа (чтобы цифра не липла к краю синего поля), эталон Ренаты:
  после правого `<w:tab/>` идут по порядку:
  1) ран `<w:t> .</w:t>` цвет FFFFFF, sz 28 — белый разделитель-зазор перед бейджем;
  2) ран `<w:t> </w:t>` (пробел) с `w:shd w:fill="1C6083"` и color 1C6083, sz 34 bold —
     невидимая ЛЕВАЯ подложка-отступ внутри синего поля;
  3) поле PAGE: color FFFFFF, `w:shd w:fill="1C6083"`, sz 34 bold — сама цифра;
  4) ран `<w:t>.</w:t>` тех же свойств — точка после числа.
  Так бейдж выглядит аккуратно на всех страницах (подвал один на весь документ).
- Header: водяной знак + логотип GG.

## Сборка и ОБЯЗАТЕЛЬНАЯ визуальная проверка
Сложное оформление править прямой правкой OOXML через lxml (python-docx не умеет тени/скругления/водяные знаки). Распаковка/запаковка: `unzip` / `zip -X -9` ([Content_Types].xml первым). После сборки проверять ВИЗУАЛЬНО, не на глаз по XML:
1. Экспорт PDF через установленный Word (AppleScript): `osascript` → `tell application "Microsoft Word"` → open / `save as ... file format format PDF` / `close ... saving no`. Оборачивать в `perl -e 'alarm 200; exec @ARGV' osascript ...` от зависаний.
   ВНИМАНИЕ: стартовый экран Word блокирует экспорт. Снять «Show Start screen when opening» или нажать Cancel.
2. Номера страниц для оглавления: venv с `pypdf`, искать текст заголовка по страницам (`PdfReader(...).pages[i].extract_text()`). Сначала собрать с токенами `@@P1@@…`, узнать реальные страницы, подставить.
3. Рендер любой страницы: pypdf вытащить одну страницу в отдельный PDF → `qlmanage -t -s 1600 -o out page.pdf` → PNG → посмотреть. Так проверять плашки, тени, цвета таблиц, центровку водяного знака на стр. 2+ (qlmanage по .docx даёт только стр. 1).
